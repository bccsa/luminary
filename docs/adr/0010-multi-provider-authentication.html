ADR 0010

Multi-Provider Authentication

Luminary · architecture and walkthrough

How login, identity linking, and group access now work across many OIDC providers.

docs/adr/0010-multi-provider-authentication.html

Why multi-provider?

Before

  • A single Auth0 tenant wired in via .env (JWT_SECRET, JWT_MAPPINGS).
  • One config baked into every deployment.
  • To add a provider: redeploy; to change claims: redeploy.
  • No CMS visibility; admins had to edit env files.

Now

  • Each provider is a CouchDB document of type authProvider.
  • CMS admins create, edit, and scope providers without touching infra.
  • Multiple providers coexist — users pick one at login.
  • Claim paths, audience, clientId, label, icon — all data-driven.

Three document types do the work

Doc typePurposeOwner
authProvider OIDC config: domain, audience, clientId, claim paths, label/icon. Admins (CMS)
autoGroupMappings Rules that assign groups on login based on JWT claims. Admins (CMS)
user Master record for a human; stores providerId + externalUserId. Auto-linked on first login + admin-editable

All three types go through the normal Luminary sync / CouchDB / ACL pipeline — no special treatment.

Client-side login flow

  1. User hits the provider-selection modal → picks provider P.
  2. Client reads P from Dexie, builds Auth0 options (domain, clientId, audience) dynamically, redirects.
  3. Auth0 redirects back with code+state; SDK exchanges for an access token.
  4. Client opens the socket with { token, providerId: P._id }.
  5. Server runs the 3-phase identity pipeline and returns an accessMap.
Auth0 localStorage keys (@@auth0spajs@@::<clientId>::*) plus the provider doc in Dexie form the client's "who am I currently logged in as" state. Clearing both is how logout + provider-switch reset the session.

Phase 1 Global default groups

  • Fetches all autoGroupMappings docs with no providerId.
  • Their groupIds are assigned to every session — authenticated users and guests alike.
  • Typical use: group-public-users → read-only view on published content.
  • Cached in-process; invalidated on any change to the relevant docs.

Anonymous users stop at this phase — no provider, no token, just defaults.

Phase 2 Provider & token verification

  1. Look up the authProvider doc by the submitted providerId.
  2. Fetch the tenant's JWKS from https://<domain>/.well-known/jwks.json.
  3. jwt.verify with: audience, issuer (https://<domain>/), RS256, signature.
  4. Check azp (or client_id) matches provider.clientId. — blocks token-reuse across providers with the same domain.
  5. Evaluate per-provider autoGroupMappings against the JWT payload → dynamic groups.
JWKS clients and provider docs are cached per-domain / per-providerId. CouchDB change events invalidate both caches automatically on edit/delete.

Phase 3 Identity linking

Find the master user doc, then compute static groups.

OrderLookupPurpose
1{ type: "user", userId: externalUserId }Deprecated legacy field — still honoured for old tenants.
2{ type: "user", externalUserId }The primary link, auto-backfilled on first login.
3{ type: "user", email }First-login fallback; also the bridge when a user has different subs across providers.

If none match → user gets defaults + dynamic, no static groups.

The backfill guard

if (externalUserId && !primaryUser.externalUserId) {
    primaryUser = { ...primaryUser, externalUserId, providerId };
} else if (!primaryUser.providerId && providerId) {
    primaryUser = { ...primaryUser, providerId };
}
  • First-time login via any provider → backfill sets externalUserId + providerId.
  • Already linked to provider A, logging in via B? → no overwrite.
  • Reason: prevents account takeover — a JWT from an untrusted IdP can't silently re-link an account via shared email.
  • Admin must re-assign the provider explicitly from the CMS.

Per-provider group scoping

Users can have multiple user docs (one per provider) — each contributes groups only when matched.

const appliesToCurrentProvider = (d) =>
    !d.providerId || d.providerId === providerId;

allMemberOf = emailUserDocs
    .filter(appliesToCurrentProvider)
    .flatMap((d) => d.memberOf ?? []);

Cache invalidation — server

All caches self-invalidate on the CouchDB change feed:

CacheInvalidated when
providerCache (by _id)provider doc update or delete
jwksClients (by domain)provider domain change or delete
autoGroupMappingsCache (by providerId)mapping update or delete
defaultGroupsCacheany provider-less mapping change

No TTLs; freshness is event-driven.

Cache invalidation — client

  • Logout: clearAuth0Cache() wipes @@auth0spajs@@::* and a0.spajs.* synchronously before the Auth0 redirect, so an interrupted logout can't leave stale provider state behind.
  • Server says "provider not found": client deletes the stale authProvider doc from Dexie, clears Auth0 cache, opens the provider selection modal. No infinite retry loop.
  • Server says "token invalid": client deletes its Dexie provider doc, lets the sync pipeline repopulate with the fresh config, then retries the login.

Discriminated auth-failure reasons

Socket connect_error payload now carries { type: "auth_failed", reason }:

ReasonTriggerClient reaction
provider_not_foundconfigured providerId no longer existsevict Dexie doc + open modal
token_invalidsignature / audience / issuer / azp mismatchevict Dexie doc + retry (Dexie re-syncs)
unsetanything else (e.g. token expired)retry with last provider
The generic message stays "Invalid authentication token" — reason codes are internal only, so no error-oracle for attackers.

Operator-facing diagnostic hint

When SigningKeyNotFoundError fires, the warn log includes:

Likely causes: (a) the access token was issued by a different Auth0 tenant than provider.domain, or (b) the token is an access token that doesn't carry the configured custom claims because your post-login action only called api.idToken.setCustomClaim (add api.accessToken.setCustomClaim too).

This is the single most common Auth0 misconfig — the rule puts claims on the ID token only, and the API (which verifies access tokens) silently gets default groups.

CMS — Auth Provider form

  • Label, domain, clientId, audience — all validated inline (hostname shape, URL shape, 8+ char clientId).
  • memberOf ("Managed by") required — server DTO enforces @ArrayNotEmpty. Red inline hint when empty.
  • Custom claim paths (externalUserId, email, name) — optional, for namespaced Auth0 claims.
  • Icon + label styling for the login screen.

Access control: only members of Managed by groups can view/edit this provider in the CMS.

CMS — Auto Group Mappings form

  • Managed by — ACL on the mapping doc itself.
  • Groups to grant on login — what the user actually receives when they match.
  • Conditions (AND): authenticated, claim equals, claim in [list].
  • No provider = global default; with provider = scoped to that login flow.
Both selectors disable save when empty, with inline red helper text explaining why — no silent no-ops.

CMS — User form

  • Admin picks the provider the user is pinned to (or "Any provider — match by email").
  • externalUserId is not admin-editable — the server backfills it from the token on first login.
  • Server-side uniqueness check on the (providerId, externalUserId) pair prevents duplicate identities at the API boundary.
  • Sidebar navigation collapsed into a single Access menu (Users, Groups, Auto Group Mappings, Auth Providers).

add-auth-provider.ts — the setup script

  • Interactive: 1) Add provider, 2) Modify provider, 3) Configure defaults, 4) Fix group ACLs, 5) Create super-admin, 6) Full setup.
  • Namespace prompt: optional URL prefix; auto-prefills claim-path defaults for Auth0 tenants with custom namespaced claims.
  • Access-token warning: surfaced whenever a namespaced claim path is configured.
  • Scoped backfill: default is "only users without a providerId"; overwriting all users requires typing overwrite.
  • Uniqueness guards: super-admin creation blocks duplicate (providerId, externalUserId) and warns on duplicate emails.

Script writes directly to CouchDB (bypasses API validation), so these guards are replicated locally.

Gotchas worth knowing

  • Access tokens vs ID tokens. Auth0 rules/actions must set custom claims on api.accessToken — the API never sees the ID token.
  • Numeric sub. Mango selectors are type-strict; the service coerces externalUserId to a string before lookup to dodge the silent mismatch.
  • Namespaced claim paths. Include the full URL (https://app.bcc.africa/metadata.personId), not just the path.
  • Multiple user docs per email. Groups are unioned within a provider; scoping filters out cross-provider bleed.
  • Legacy users (userId field). Still matched by Action 1; script's backfill migrates them to externalUserId.

Known tech debt / follow-ups

  • Orphan @@auth0spajs@@::<oldClientId>::* localStorage keys after a provider is deleted — harmless, worth a janitor eventually.
  • Dead JWT_SECRET / JWT_MAPPINGS env vars from the old single-provider flow — safe to remove.
  • Auto-clearing old Auth0 tokens when an admin swaps a provider's clientId.
  • Better CMS surfacing of the "this user doc is for a different provider" case (currently the per-provider filter just silently drops its groups).
